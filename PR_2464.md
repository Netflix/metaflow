## Fix `@resources` memory/cpu/disk silently ignored when less than `@kubernetes` default

**Fixes #2464**

### Problem

When using `@resources` with `--with kubernetes` (or alongside `@kubernetes`), resource values smaller than the `@kubernetes` built-in defaults were silently ignored:

```python
@resources(cpu=1, memory=256)  # user explicitly wants 256 MB
@step
def start(self):
    ...
```

```bash
python flow.py run --with kubernetes
# Pod is allocated 4096 MB (the @kubernetes default), not 256 MB
# No warning is emitted
```

**Root cause:** `step_init()` merged `@resources` into `@kubernetes` attributes using `max()`. Since the `@kubernetes` default for memory is `4096`, `max(4096, 256) = 4096` — even when the user never touched `@kubernetes(memory=...)` at all. This also resolves the long-standing `TODO: Fix https://github.com/Netflix/metaflow/issues/467`.

### Fix

Before taking `max()`, check if the `@kubernetes` attribute is still at its **built-in default** (i.e., the user never explicitly passed it to `@kubernetes(...)`). If so, honour the `@resources` value directly.

```
If @kubernetes value == its default  AND  @resources provides a value
    → use the @resources value  (no max())
Else
    → use max() as before
```

This preserves existing semantics when both decorators have **explicit** values:

| `@resources` | `@kubernetes` explicit | Result |
|---|---|---|
| `memory=256` | *(default 4096)* | **256** ✅ |
| `memory=8192` | *(default 4096)* | **8192** ✅ |
| `memory=256` | `memory=1024` | **1024** (explicit kube wins — it's larger) |
| `memory=8192` | `memory=1024` | **8192** (resources wins — it's larger) |

The fix applies to **all** numeric resource attributes (`cpu`, `memory`, `disk`), not just memory.

### Changes

- **`metaflow/plugins/kubernetes/kubernetes_decorator.py`** — Replace blind `max()` in `step_init()` with a default-aware merge. Remove stale `TODO: Fix #467` comment.
- **`test/unit/test_kubernetes.py`** — Add 7 parametrized regression tests covering all merge scenarios.
